# План A/B-теста

Документ описывает постановку A/B-теста для сравнения двух версий модели прогнозирования дефолта по кредитным картам.

## 1. Контекст и цель

В сервисе развёрнуты две модели:
- **v1 — LogisticRegression** (контрольная группа, текущая модель в проде)
- **v2 — RandomForestClassifier** (тестовая группа, новая модель-кандидат)

Цель теста — понять, какая модель полезнее для банка: та, что ловит больше дефолтов (высокий Recall), или та, что реже беспокоит нормальных клиентов отказом (высокий Precision).

## 2. Гипотезы

**Нулевая гипотеза (H0):** доля пойманных дефолтов в группах A (v1) и B (v2) одинакова.

```
H0: recall_A = recall_B
```

**Альтернативная гипотеза (H1):** доли различаются.

```
H1: recall_A ≠ recall_B
```

Тест двусторонний, потому что заранее не известно, какая модель окажется лучше.

## 3. Разделение трафика

- **Способ:** случайное распределение 50/50 при каждом запросе к `/predict` без параметра `version`.
- **Реализация:** в `app/api.py` при отсутствии `?version=v1|v2` используется `random.choice(['v1', 'v2'])`.
- **Единица рандомизации:** отдельный запрос. В реальном проде стоило бы разделять по `client_id`, чтобы один клиент всегда попадал в одну группу (stickiness), но для учебного проекта этого достаточно.

## 4. Продолжительность теста

- **Минимум 2 недели** — чтобы стабилизировались кумулятивные метрики и учёлся недельный сезонный цикл платежей.
- **Минимальный размер выборки:** ориентировочно по 1000 клиентов-дефолтников в каждой группе для достижения мощности теста 80% при α=0.05 и minimum detectable effect (MDE) около 5 п.п. разницы в recall.

## 5. Метрики

### Основная метрика: Recall по классу дефолта

```
recall = TP / (TP + FN)
```

Доля реальных дефолтов, которые модель правильно обнаружила. Выбрана как основная, потому что **пропущенный дефолт стоит банку больше денег** (вся сумма долга), чем ложное срабатывание (упущенная прибыль по одному клиенту).

### Дополнительная метрика: Precision по классу дефолта

```
precision = TP / (TP + FP)
```

Доля предсказанных дефолтов, которые оказались правдой. Важна, потому что слишком агрессивная модель с низким Precision будет отказывать множеству нормальных клиентов — банк потеряет клиентов и прибыль.

**Почему выбраны именно эти две метрики:** они напрямую описывают два типа ошибок модели (FN и FP), которые по-разному сказываются на бизнесе.

### Бизнес-метрика: предотвращённые финансовые потери

```
savings = TP * avg_loss_per_default
```

Где `avg_loss_per_default` — средний убыток банка на один дефолт (условно 50 000 NT$). Показывает ценность модели в деньгах.

## 6. Статистический анализ

### Z-тест для пропорций

Сравниваем доли пойманных дефолтов (recall) между группами. Применяем `statsmodels.stats.proportion.proportions_ztest`:

```python
from statsmodels.stats.proportion import proportions_ztest

alpha = 0.05
_, p_value = proportions_ztest(
    count=[tp_a, tp_b],
    nobs=[n_defaults_a, n_defaults_b],
    alternative='two-sided'
)
```

- **Уровень значимости:** α = 0.05
- **Интерпретация:** если p-value ≤ 0.05, отвергаем H0 и делаем вывод о значимой разнице между моделями

### Доверительный интервал разности пропорций

Строим 95% ДИ для `recall_B - recall_A` по формуле из курса:

```python
def diff_proportion_conf_interval(x_p, n, gamma=0.95):
    alpha_ci = 1 - gamma
    diff = x_p[1] - x_p[0]
    z_crit = -norm.ppf(alpha_ci / 2)
    eps = z_crit * (x_p[0] * (1 - x_p[0]) / n[0]
                  + x_p[1] * (1 - x_p[1]) / n[1]) ** 0.5
    return diff - eps, diff + eps
```

## 7. Критерии успеха

Новая модель v2 заменяет v1, если **одновременно** выполняются:

1. **Статистическая значимость:** p-value < 0.05
2. **Направление эффекта:** нижняя граница 95% ДИ разности `recall_B - recall_A` > 0 (то есть v2 статистически уверенно выше v1)
3. **Precision не упал критично:** precision_B ≥ 0.9 * precision_A (допустимая просадка не более 10%)

Если хотя бы одно условие не выполнено — v1 остаётся в проде.

## 8. Риски и ограничения

- **Новизна модели:** в первые дни v2 может показывать худшие метрики из-за отсутствия адаптации пользователей (к скорингу это применимо слабо, но стоит учитывать при интеграциях).
- **Множественные сравнения:** если тестируется несколько метрик одновременно, нужна поправка Бонферрони: α_adj = 0.05 / k.
- **Сезонность:** если тест короче 2 недель, выходные и будние дни могут дать смещённую картину.

## 9. Влияние на архитектуру сервиса

A/B-тест встроен прямо в Flask API:

- Параметр `?version=v1|v2` позволяет принудительно выбрать модель (для логирования, отладки, внутренних тестов).
- Без параметра — случайное распределение 50/50 (`random.choice` в `app/api.py`).
- Каждый запрос логируется в JSON с указанием версии — это даёт возможность собрать статистику по группам без дополнительной инфраструктуры.
- В production логи из контейнера собирал бы ELK-стек (Logstash → Elasticsearch → Kibana), а уже из Kibana можно было бы считать кумулятивные recall/precision по группам в реальном времени.

## 10. Фактические результаты на тестовой выборке

Запуск `python ab_test/ab_analysis.py` на отложенной выборке (6000 клиентов, из них 1327 дефолтников):

| Группа | Модель | Recall | Precision | F1 |
|--------|--------|--------|-----------|-----|
| A | v1 LogisticRegression | 0.612 | 0.367 | 0.461 |
| B | v2 RandomForestClassifier | 0.354 | 0.647 | 0.448 |

- **p-value z-теста:** ≈ 0.0 (разница значима)
- **95% ДИ разности recall (B - A):** (-31.0%, -20.6%)
- **Вывод:** ДИ целиком отрицателен — **v1 статистически значимо ловит больше дефолтов, чем v2**. По основной метрике v1 побеждает.

Однако Precision у v2 выше почти вдвое — если стоимость ложного срабатывания высока (отказ хорошему клиенту → уход к конкуренту), v2 может оказаться выгоднее. Финальное решение зависит от бизнес-расчёта `avg_loss_per_default` vs `avg_profit_per_client`.
