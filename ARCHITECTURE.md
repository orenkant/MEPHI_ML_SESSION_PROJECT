# Архитектура сервиса

## Монолит vs Микросервисы

Для данного учебного проекта выбран **монолитный подход**.

**Почему монолит:**
- Один сервис — один процесс. Проще запустить, отладить, задеплоить.
- Нет сетевого взаимодействия между компонентами — меньше точек отказа.
- Для одной модели и небольшой нагрузки монолит достаточен.
- Меньше инфраструктуры: один Dockerfile, один порт, один контейнер.

**Когда перейти на микросервисы:**
- Нужно деплоить модели независимо друг от друга.
- Разные компоненты требуют разных ресурсов (CPU/GPU).
- Несколько команд работают над разными частями системы.
- Нагрузка на отдельные части растёт по-разному.

В production-банке имело бы смысл выделить:
- сервис предобработки данных
- сервис инференса модели
- сервис логирования и мониторинга

## Логирование

Сервис логирует каждый запрос к `/predict` в JSON-формат:

```json
{"time": "2026-05-01 13:00:00", "level": "INFO", "message": "{\"version\": \"v1\", \"prediction\": 1, \"probability\": 0.67}"}
```

В production-среде логи собираются через **ELK-стек**:
- **Logstash** — сбор логов из контейнеров
- **Elasticsearch** — хранение и индексирование
- **Kibana** — визуализация и дашборды

## Концепт RabbitMQ

В гипотетическом сценарии масштабирования брокер сообщений нужен для:

1. **Батч-предсказания**: вместо 1000 синхронных HTTP-запросов клиент отправляет одно сообщение в очередь, воркер обрабатывает батч и возвращает результат.
2. **Асинхронное логирование**: сервис публикует лог-сообщения в очередь, отдельный консьюмер пишет их в базу/ELK без блокировки основного потока.
3. **Retry-логика**: если модель временно недоступна, сообщение остаётся в очереди и обрабатывается позже.

Пример схемы:
```
Client → POST /predict → RabbitMQ queue → ML Worker → результат в базу
```

## uWSGI + NGINX

В production Flask не запускают напрямую (`python app.py`), потому что встроенный сервер однопоточный и не рассчитан на нагрузку.

- **uWSGI** — WSGI-сервер, запускает несколько воркеров Flask параллельно.
- **NGINX** — reverse proxy перед uWSGI: балансировка, SSL-терминация, раздача статики, защита от медленных клиентов.

В данном проекте используется **Gunicorn** — более простая альтернатива uWSGI с аналогичной функцией.

## ONNX-ML

Для ускорения инференса модель можно конвертировать в формат ONNX:

```python
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType

initial_type = [('float_input', FloatTensorType([None, 23]))]
onnx_model = convert_sklearn(pipeline_v1, initial_types=initial_type)
with open('models/model_v1.onnx', 'wb') as f:
    f.write(onnx_model.SerializeToString())
```

Преимущества: быстрее инференс (особенно на CPU), портируемость между фреймворками.

## DVC и MLflow

**DVC (Data Version Control):**
- Версионирует большие файлы данных и модели (аналог git для бинарных файлов).
- Хранит файлы в S3/GCS/локально, а в git — только указатели.
- В данном проекте пригодился бы для версионирования `UCI_Credit_Card.csv` и `model_v1.pkl`.

**MLflow:**
- Отслеживает эксперименты: параметры, метрики, артефакты.
- Позволяет сравнивать запуски и воспроизводить лучший результат.
- В данном проекте — логировать F1/Precision/Recall при каждом запуске `src/train.py`.

## Бизнес-метрики

Помимо F1/Precision/Recall важны метрики, понятные бизнесу:

**1. Предотвращённые финансовые потери (NT$)**

```
savings = TP * avg_loss_per_default
```

Где `TP` — правильно предсказанные дефолты, `avg_loss_per_default` — средний убыток банка на один дефолт. Показывает реальную ценность модели в деньгах.

**2. Доля одобренных заявок при том же уровне риска**

При фиксированном пороге вероятности (например, 0.5) считаем долю клиентов, которых модель не пометила как рискованных. Увеличение этой доли означает, что банк одобряет больше кредитов без роста дефолтов.

```
approval_rate = (TN + FN) / total
```

## A/B-тест: постановка

**Гипотеза:** RandomForest (v2) лучше обнаруживает дефолты, чем LogisticRegression (v1).

**Разбиение трафика:** случайное 50/50 при каждом запросе без указания версии.

**Продолжительность:** минимум 2 недели (для стабилизации кумулятивных метрик).

**Основная метрика:** Recall по классу дефолта (доля пойманных дефолтов).

**Дополнительная метрика:** Precision по классу дефолта. Выбран Precision, потому что ложное срабатывание (отказ нормальному клиенту) тоже стоит банку денег — потеря клиента.

**Статистический анализ:**
- Z-тест для пропорций (двусторонний), α = 0.05
- 95% доверительный интервал разности recall
- Критерий успеха: p-value < 0.05 И нижняя граница ДИ > 0 (v2 выше v1)
